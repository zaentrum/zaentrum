> you own your library

zaentrum is content-neutral. you point it at files that are already on disk; it catalogs and streams them. there are no built-in downloaders and no indexer integrations — how the files got there is out of scope, by design.

> the platform

roughly sixteen services — one image on your laptop, or a full cluster. same manifests, either way.

one command

a single container runs a full kubernetes in-process: web, admin, catalog, transcode, package, streaming. one image, one port, no external dependencies.

bundled backends

postgres, valkey and a kafka broker ship inside the container. nothing external to stand up before you can browse your first title.

event-driven pipeline

scan → enrich → analyze → transcode → package. every stage handoff is a kafka event keyed by item. optional gpu (nvenc) acceleration; software by default.

clean clients

web, phone/tablet and tv — skins over one shared core. point any of them at your server with add server: an ip, a .local name, or a real domain.

scales out

the same platform runs on any kubernetes cluster, reconciled by an operator from a single custom resource — with day-2 scaling and channel auto-updates.

your identity

run the bundled keycloak, federate your existing provider, or delegate to an external oidc provider — chosen in the first-run setup wizard.

> get started

two paths, same platform. start on your laptop; move to a cluster when you outgrow it.

one box

everything in a single container.

bash
$ docker run -d --privileged -p 80:80 \
    --name zaentrum \
    ghcr.io/zaentrum/zaentrum:latest
$ open http://zaentrum.localhost

the first-run wizard names it, sets identity, and points it at your library. self-hosting guide →

on kubernetes

the same platform, operator-managed.

bash
$ kubectl apply -k \
    github.com/zaentrum/zaentrum-operator/config
$ kubectl apply -f zaentrum.yaml   # your Zaentrum CR

one operator reconciles ~16 services from one custom resource. operator & cr reference →
